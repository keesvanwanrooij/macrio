## Problem

Macros differ a lot between raw and prepared weight (rice, meat, pasta). “100 g rice” is ambiguous. Users want to **toggle** cooked vs uncooked macros on the **same** product, not hunt two separate catalog entries.

## Goal

Declare cook applicability and, when both matter, store **both** nutrition sets so the UI can switch without forking the product identity.

## Recommended data model (inspect outcome)

**Prefer one version row with dual nutrition**, not two products:

| Approach | Pros | Cons |
|---|---|---|
| **A. Dual macros on same version (chosen)** | One barcode/product; toggle is UI; versions still immutable snapshots of both blocks | Slightly wider schema; create form longer |
| B. Two linked products (`paired_product_id`) | Simple single-macro rows | Duplicate barcodes/names; likes split; merge pain |
| C. Two versions of same product (v1 cooked, v2 uncooked) | Reuses versions | “Current version” likes fight; toggle ≠ community edit |

### Schema sketch (A)

On `product_versions`:

- `cook_state` text: `cooked` \| `uncooked` \| `not_applicable` \| `both`
  - `both` = dual macros present; UI shows a cooked/uncooked toggle
  - single state = only primary macros used (today’s columns)
- When `cook_state = 'both'`:
  - Keep existing `kcal_100g` / carbs / protein / fat as the **default** state (usually uncooked for staples, or cooked for ready meals - pick rule)
  - Add `alt_kcal_100g`, `alt_carbs_100g`, `alt_protein_100g`, `alt_fat_100g` for the other state
  - Add `alt_cook_state` text (`cooked` \| `uncooked`) naming what the alt block is
- Diary `log-entry`: user picks state; **snapshot** stores the macros actually used (already true today)
- Suggest-edit / new version: copies both primary + alt blocks; **30-day owner edit rule** (v1.2) applies to the whole version the same as other foods

### Why not put cook_state only on `products`

Community edits change nutrition; cook pairing is nutrition. Versions stay the source of truth.
