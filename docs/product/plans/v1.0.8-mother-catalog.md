## Problem

Before public launch we need a catalog model that:

- Lets anyone create food when a barcode is missing (already works)
- Groups versions by barcode under one **mother** (`products` row)
- Gives users ownership of **versions**, not of barcodes or mothers
- Lets the founder (admin account) name mothers and seed the DB without blocking the app when a mother title is empty
- Controls DB growth (cap versions per owner; trim unused)

## Locked model

### Layers

| Layer | Table (today) | Role |
|---|---|---|
| **Mother** | `products` | Catalog identity. Optional barcode. Optional canonical display name (admin can set later). No user “owns” the mother for barcode purposes. |
| **Version** | `product_versions` | Nutrition / allergens / portions / title as the author wrote it. **Owner** = creator. Visibility public \| private lives **here** (not on mother). |

We already have two tables. **Do not invent a third “versions” table.** Evolve `products` + `product_versions` (and RLS/RPCs).

### Scan / create

1. User scans barcode → if mother exists → show versions like today (ranking below). Suggest improvement = new version under that mother (subject to caps).
2. Barcode missing → create flow → **version 1** under a mother for that barcode (create mother if needed). Creator owns the version, not the mother.
3. **No-barcode mothers** (generic / staple foods): allowed. Versions link by `product_id`. Not auto-grouped with anything else until a barcode is attached.
4. **Quick-add:** diary-only (macros + name on the entry). **Not** a catalog mother/version. Private to the user’s diary. Do not insert into `products` / `product_versions`.

### Display names / discovery

- **Search, recents, favorites:** find and show **versions**. List rows use each **version title** (as the author wrote it). People do not search “mothers” as the primary result.
- **Mother name:** shown when **browsing versions** on the product/mother screen (header / grouping label), not as the search result title.
- If mother name is empty in the table: on that browse screen, show the **most liked version’s name** as the mother label (ties: completeness → newest).
- Admin can set a canonical mother name later; version titles stay unchanged.

### Ranking (default / top of version list)

1. Most likes  
2. Completeness (see completeness plan)  
3. Most recent  

### Visibility

- **On versions only** (migrate away from `products.visibility` as the catalog gate).
- Public versions visible to everyone (subject to RLS).
- Private versions: only the owner (and maybe admin for support).
- Seed / staple content: versions (and mothers as needed) attributed to the **reserved admin account**.

### Barcode link / unlink (version-driven grouping)

- Mothers with a barcode group all versions that belong to that barcode identity.
- Owner of a version may **add a barcode** → version (re)links into that barcode’s mother as a **new version** under that mother (merge path). If mother for that barcode did not exist, create it.
- Owner may **remove barcode** from their version → unlink from that barcode mother (version stays owned; lands on / under a no-barcode mother per RPC rules - specify in impl).
- One public barcode → one mother (keep uniqueness for public barcode identity).

### Caps and hygiene (DB size)

- **Max 3 versions per owner per mother.** Creating a 4th deletes/archives that owner’s **oldest** version on that mother (see safety below).
- System trim: archive/hide old versions that are low-like and unused for a long time (safe job; prefer soft-delete).
- Founder/admins can rename mothers and clean bad rows from the backend; later: assign more admin users.

### Admin / reserved usernames

- Reserved username list (e.g. `keesvanwanrooij`, `macrio`, `admin`, …) - cannot be taken at sign-up; only creatable from backend.
- Admin account owns seed mothers/versions.
- Staples = normal mothers seeded under admin (same table as user-generated mothers). No separate “parent” privilege in v1.0 beyond admin seed + rename.

### Suggest improvement

Anyone can open a version and suggest a better one (new version under same mother), within the 3-per-owner cap and visibility rules.

## Safety: max 3 versions / deletes (DB writes)

**Yes, this can be safe** if we avoid hard-deleting rows that diary still points at:

| Approach | Recommendation |
|---|---|
| Diary integrity | Diary already **snapshots macros** on the entry. Keep `product_version_id` nullable or point at a tombstone; never rewrite historical kcal. |
| Cap eviction | Prefer **soft-delete** (`archived_at` / `replaced_by`) over hard `DELETE`. Hard delete only when **zero** diary refs and not current default. |
| Concurrent writes | Enforce cap in an **RPC** (transaction: count owner versions → archive oldest → insert new). Do not rely on client-only checks. |
| Likes | Cascade or keep likes on archived versions but exclude from ranking. |

Client-only “delete oldest then insert” is **not** safe under concurrency; RPC + soft-delete is.

## Schema direction (impl notes)

- Keep `products` (mothers) + `product_versions`.
- Move catalog visibility to `product_versions.visibility` (migrate from `products.visibility`).
- Mother: optional `name_nl` / `name_en` (canonical overwrite); barcode unique for public identity as today.
- Version: owner (`edited_by` / `created_by`), title fields, optional barcode attach fields **or** mother barcode + explicit reassign RPCs (choose one consistent model in migration).
- Completeness helper shared with v1.0.10 ranking.
- Reserved usernames check on profile create + seed admin user.
- RPCs: `suggest_version`, `attach_version_barcode` (merge into mother), `detach_version_barcode`, `create_product_full` aligned with mother rules; admin rename mother.

## Acceptance criteria

- [ ] Scan known barcode → existing mother + ranked versions; suggest improvement adds a version
- [ ] Scan unknown → create mother + version 1; user owns version only
- [ ] Search/recents show **versions** (version titles); mother name only on version-browse screen
- [ ] Mother name empty → browse header uses most liked version’s name (ties: completeness → newest); admin can set mother name later
- [ ] Version titles unchanged by mother rename
- [ ] Quick-add never creates catalog rows
- [ ] No-barcode mother works; versions link by product id; barcode attach merges into barcode mother
- [ ] Visibility on versions; seed under reserved admin account
- [ ] Max 3 versions per owner per mother enforced in RPC with safe archive
- [ ] Reserved usernames blocked in app sign-up
- [ ] Docs / DATA_MODEL / NAMING updated to mother language

## Open questions (small)

## Test plan

- [ ] Search shows version titles; opening a hit → mother browse shows mother name (or most-liked fallback if unset)
- [ ] Two users, same new barcode: second user sees first mother, does not create a duplicate mother
- [ ] Fourth version by same owner archives their oldest; diary entries that used archived version keep macros
- [ ] Attach barcode on no-barcode version → appears under barcode mother
- [ ] Detach barcode → unlinked; mother with no remaining versions handled cleanly
- [ ] Reserved username rejected at sign-up
- [ ] Quick-add: diary row only, no product created
